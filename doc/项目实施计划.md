# 《法言白话》项目实施计划

## Summary

- 目标：基于当前 HarmonyOS Empty Ability 模板工程，按文档实现《法言白话》首版，覆盖客户端与同仓后端两部分。
- 首版主链路：导入合同 -> 分析中反馈 -> 结果总览 -> 对照阅读 -> 为后续多设备接续保留能力接入点。
- 实施策略：先完成客户端壳工程和统一数据契约，再做前后端 Mock 可联调闭环，随后替换为真实接口与系统能力。
- 执行约束：后续实施必须遵循 [项目实现步骤](file:///Users/mac/DevEcoStudioProjects/ContractScanner/doc/项目实现步骤.md) 的顺序，按最小步骤推进，并在每一步完成后回写状态。

#### 改动内容

- 在当前仓库新增 `server/`，按蓝图建立 TypeScript 后端目录。
- 首轮先提供可运行的接口桩与 Mock 分析链路，固定三个接口：
  - `POST /v1/contracts/analyze`
  - `GET /v1/tasks/{taskId}/status`
  - `GET /v1/tasks/{taskId}/result`
- 用统一 schema 约束请求和响应，保证客户端服务层可以稳定对接。
- 将 OCR、文本切分、LLM 分析等先抽象成可替换模块，首版先用 Mock/规则生成结果，再逐步替换真实实现。

#### 这样做的原因

- 用户已明确要求前后端都开发。
- 当前仓库无后端基础，直接上真实 OCR/LLM 会把不确定性放到最前面，不利于快速形成可演示闭环。
- 先起服务骨架和 Mock 结果，可尽快完成“客户端可跑、后端可回、协议已定”的第一阶段联调。

## Assumptions & Decisions

- 决策 1：本轮计划覆盖前后端，不只做客户端。
- 决策 2：后端与客户端同仓组织，新增 `server/` 目录。
- 决策 3：先完成“可演示闭环 + Mock/规则结果”，再替换为真实 OCR/LLM。
- 决策 4：客户端继续保持单 module、单 Ability 结构，不在首轮拆多模块。
- 决策 5：页面与状态管理遵循蓝图目录，不额外引入重型第三方框架。
- 假设 1：当前环境可继续使用 DevEco Studio/Hvigor 构建 ArkTS 客户端。
- 假设 2：后端首轮可使用 Node.js + TypeScript 在本地运行；具体依赖以仓库现状为准，优先轻量。
- 假设 3：拍照/相册/文档上传入口在首版允许先以占位流程或文件选择能力跑通，不要求第一步就完成所有系统能力的真接入。
- 假设 4：真实 OCR/LLM 凭据、部署地址、生产级隐私策略不在首轮阻塞项内，但接口结构需预留。

## Verification Steps

### 阶段 A 验证

- 确认 Ability 入口能加载业务首页而不是模板页。
- 确认新增页面均已在 `main_pages.json` 注册。
- 确认目录骨架、模型、公共模块、路由封装均可编译通过。
- 确认首页可跳转到导入页。

### 阶段 B 验证

- 确认客户端可完成“选择合同 -> 发起分析 -> 查看状态 -> 查看结果 -> 进入对照阅读”。
- 确认后端三个基础接口可本地响应。
- 确认失败态覆盖导入失败、分析失败、结果缺失三类场景。
- 确认最近分析记录可缓存并在首页展示。

### 阶段 C 验证

- 确认应用接续可从手机流转到平板或 PC 查看同一结果。
- 确认互动卡片能展示最近分析摘要并跳转结果页。
- 确认隐私防窥开关能影响敏感页面展示策略。

### 验收标准

- 主链路完整可演示。
- 前后端协议固定且客户端已完成对接。
- 至少三项鸿蒙特性进入可展示状态。
- 风险项能回溯到原文与白话解释。
- [checklist.md](file:///Users/mac/DevEcoStudioProjects/ContractScanner/doc/specs/checklist.md) 中与首版实现相关项可逐项核验。
