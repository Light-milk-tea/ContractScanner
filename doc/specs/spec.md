# 《法言白话》项目规划 Spec（优化版）

## Why
应用创新赛道强调四件事：鸿蒙原生、AI 融合、场景实用、体验亮点，并建议集成 3 个及以上鸿蒙特性。  
《法言白话》选题痛点明确，但为了保证赛期可交付，需要把方案进一步收敛为“可基于 DevEco Studio 基础模板快速搭建、可在评审现场稳定演示”的实现路径。

## What Changes
- 固定首版目标：面向非法律专业用户，提供“合同白话解读 + 风险提示 + 签署前摘要”。
- 固定主链路：导入合同 -> 分析中反馈 -> 结果总览 -> 对照阅读 -> 多设备接续查看。
- 固定技术形态：HarmonyOS 原生应用（ArkTS/ArkUI，Stage 模型），不以纯 Web 承载核心流程。
- 固定工程起点：基于 DevEco Studio 的 Empty Ability 基础模板演进，先跑通单模块、单入口、单主流程。
- 固定差异化特性优先级：应用接续/跨设备互通、互动卡片、隐私防窥（至少 3 项）。
- 固定非首版边界：不做电子签署、律师咨询、合同改写、多人协作、法律意见书。

## Impact
- Affected specs: 产品范围、鸿蒙特性设计、演示路径、客户端与服务端边界、异常处理与隐私策略。
- Affected code（模板演进后）: `entry/src/main/ets/pages`、`entry/src/main/ets/components`、`entry/src/main/ets/services`、`entry/src/main/ets/model`、`entry/src/main/module.json5`、`AppScope/app.json5`。

## Technical Approach
- 使用 DevEco Studio 基础模板创建工程，保留默认生命周期与构建配置，先在模板首页挂接最小业务入口。
- 客户端采用 ArkTS + ArkUI 实现页面与交互，负责文件导入、请求发起、状态反馈、结果展示、本地缓存。
- 服务端负责 OCR、文本清洗、条款切分、风险分级、白话解释与摘要生成，并返回结构化 JSON。
- 首版默认“手机优先”，平板/PC 用于接续展示；布局支持列表-详情自适应但不做复杂多端定制。
- 通过统一数据模型承载分析结果，保证首页摘要、风险列表、对照阅读三个视图的数据一致性。
- 对关键失败场景提供明确回退：导入失败、识别失败、分析超时、结果不完整。

## DevEco Studio 落地约束
- 工程从 Empty Ability 模板起步，避免首期引入多 module 或复杂插件。
- 所有页面先使用基础组件实现可用版本，再按时间补充视觉优化。
- 鸿蒙特性接入采用“先演示可见、再完善体验”策略，确保答辩时能稳定触发。
- 依赖项遵循“够用优先”，禁止在首版引入重型三方框架导致构建不稳定。

## ADDED Requirements
### Requirement: 基于模板的原生工程起步
系统 SHALL 基于 DevEco Studio 基础模板构建首版工程，并在模板结构上增量实现业务能力。

#### Scenario: 模板工程可直接演进
- **WHEN** 团队创建新项目
- **THEN** 可通过 Empty Ability 模板直接启动开发，并在不破坏默认构建链路的前提下新增页面与服务层

#### Scenario: 模板首页接入业务入口
- **WHEN** 首次运行模板工程
- **THEN** 用户可从默认首页进入“导入合同”流程，证明业务是对模板的增量扩展

### Requirement: HarmonyOS 原生实现
系统 SHALL 以 HarmonyOS 原生应用形态实现核心功能，并使用 ArkTS/ArkUI 承载主流程。

#### Scenario: 原生页面承载主链路
- **WHEN** 用户使用《法言白话》
- **THEN** 首页、上传页、分析页、结果页、对照阅读页均由 ArkUI 原生页面实现

### Requirement: 合同导入与文本获取
系统 SHALL 支持拍照、图片导入、文档上传三种入口，并将内容转换为可分析文本。

#### Scenario: 图片合同导入成功
- **WHEN** 用户拍照或从相册导入合同图片
- **THEN** 系统完成预处理并进入分析准备状态

#### Scenario: 文档合同导入成功
- **WHEN** 用户上传 PDF 或常见文档格式
- **THEN** 系统提取正文并保持段落顺序

### Requirement: 客户端与 AI 服务协同
系统 SHALL 采用“鸿蒙客户端 + 解析/大模型服务”协同架构，保证首版稳定可演示。

#### Scenario: 返回结构化分析结果
- **WHEN** 用户提交分析请求
- **THEN** 服务返回摘要、风险等级、原文片段、白话解释等结构化结果

#### Scenario: 分析中状态可感知
- **WHEN** 分析未完成或失败
- **THEN** 客户端展示进度文案、失败原因与重试入口

### Requirement: 白话解读与风险分级
系统 SHALL 输出面向普通用户的解释文本，并提供红黄绿三色风险标记。

#### Scenario: 风险原因可理解
- **WHEN** 用户查看风险条款
- **THEN** 页面同时展示风险级别与原因说明，避免“只给结论”

### Requirement: 结果可回溯对照
系统 SHALL 提供“原文 - 白话 - 风险原因”的对照阅读，支持从摘要和风险列表跳转。

#### Scenario: 从风险项定位原文
- **WHEN** 用户点击某条风险
- **THEN** 可直接定位到对应原文片段与解释内容

### Requirement: 鸿蒙特性集成（至少 3 项）
系统 SHALL 在首版集成至少 3 项与场景强相关的鸿蒙特性，且在答辩中可现场演示。

#### Scenario: 多设备接续
- **WHEN** 用户从手机切换到平板或 PC
- **THEN** 可继续查看同一份合同的分析结果

#### Scenario: 互动卡片
- **WHEN** 用户添加桌面卡片
- **THEN** 卡片展示最近分析状态或风险摘要并支持点击跳转

#### Scenario: 隐私防窥
- **WHEN** 用户在公共场景查看敏感合同内容
- **THEN** 系统开启防窥保护降低泄露风险

### Requirement: 首版范围可控
系统 SHALL 以“小而美、闭环完整、评审可感知”为原则控制范围。

#### Scenario: 新需求评估
- **WHEN** 团队讨论新增功能
- **THEN** 只保留提升主链路演示价值的需求，其余进入二期清单
