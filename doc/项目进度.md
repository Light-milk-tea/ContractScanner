## 1. 使用规则（给后续 AI）

### 1.1 执行规则
- 每次只执行一个最小步骤，不得跨步骤提前实现后续内容。
- 执行前先阅读与当前步骤相关的现有代码和配置。
- 若当前步骤依赖前一步未完成，先停止并说明阻塞原因，不得跳步。
- 若发现实现需要调整，以当前仓库代码为准，但不得偏离蓝图中的主链路目标。

### 1.2 回写规则
- 开始执行某一步前，将该步骤状态从 `待完成` 改为 `进行中`。
- 步骤完成后，必须将状态改为 `已完成`。
- 步骤完成后，必须同步补充：
  - 完成日期
  - 改动文件
  - 验证结果
  - 备注（可选）
- 如果执行失败或被阻塞：
  - 将状态改为 `阻塞`
  - 在“备注”中写明阻塞原因
  - 不得伪造“已完成”

### 1.3 状态约定
- `待完成`：尚未开始
- `进行中`：当前正在执行
- `已完成`：代码已落地，并完成基本验证
- `阻塞`：存在前置条件缺失或环境问题，当前无法继续

### 1.4 回写示例

```md
- [x] A1 验证基础工程可运行
  - 状态：已完成
  - 完成日期：2026-07-06
  - 改动文件：无
  - 验证结果：工程入口文件可正常加载 `pages/Index`
  - 备注：确认当前项目为 Empty Ability 初始模板
```

## 2. 当前执行原则

- 先做壳工程和主链路，再做鸿蒙特性，再做答辩强化。
- 首版优先保证“导入合同 -> 分析中 -> 结果总览 -> 对照阅读”完整跑通。
- 首版允许先用 Mock 数据和占位实现前后端联调，但 Mock 仅用于演示闭环，不代表核心能力已完成。
- 《法言白话》的核心用户价值依赖真实大模型能力，白话解释、风险原因、修改建议必须通过服务端接入大模型实现，不能以纯规则或纯 Mock 作为最终交付。
- 客户端只负责文件导入、状态轮询和结果展示，不直接调用第三方大模型 API；模型调用统一收敛在服务端。
- 不提前做二期能力：电子签署、律师咨询、合同改写、多人协作、法律意见书导出。
